# AutoBiology 的原理与方法

## 1. 研究目标与问题定义

生物实验标准操作规程（standard operating procedure, SOP）通常以自然语言描述实验步骤、参数、风险和人工判断。面向生物装备研发时，仅依靠原始 SOP 难以直接得到可追溯、可验证、可去重的工程设计需求。AutoBiology 的目标是将 SOP 文本转化为结构化自动化需求表，使每条需求能够追溯到来源操作和证据字段，并区分确定性规则结论、候选推理结论和待澄清问题。

给定一个 SOP 文本 `S`，方法输出五类中间或最终产物：

- 操作表 `O`：由原始 SOP 片段原子化得到的操作集合。
- 知识超图 `G = (V, E)`：将操作、物料、参数、条件、风险、处理方式等表示为节点和操作超边。
- 需求表 `R`：按 R1-R10 分类生成的工程需求集合。
- 待澄清集合 `C`：记录缺失参数、不确定推理和 LLM 降级信息。
- 覆盖率与审阅报告 `Q`：用于评价需求对操作超边和关键角色字段的覆盖程度。

该问题可表述为一个证据约束的转换任务：

```text
F: SOP text -> (OpTable, Hypergraph, RequirementTable, CoverageReport)
```

其中每条需求 `r in R` 必须保留 `sourceOps` 和 `sourceHyperedges`，以满足从需求回溯到 SOP 操作证据的要求。没有可引用来源超边的推理结果不得进入需求表。

## 2. 方法总体框架

AutoBiology 采用五阶段流水线：

1. SOP 原子化：将原始 SOP 拆解为最小可执行操作，并抽取操作字段。
2. 知识超图构建：将操作字段归一化为节点，并以操作超边连接多角色证据。
3. 确定性需求生成：根据超图字段执行 R1-R10 直接映射和领域模式检查。
4. LLM 辅助候选推理：在配置完整时生成隐含候选需求、改写句式并辅助语义去重。
5. 覆盖率审阅：构建需求覆盖矩阵，生成验证警告、图示和可审阅报告。

这一框架的基本原则是：确定性规则层产生可解释、可复现的基础需求；LLM 层只补充候选项，并通过来源约束、去重和专家审阅降低过度推理风险。

## 3. SOP 原子化

SOP 原子化阶段将连续自然语言文本拆解为操作记录。首先，系统对 Markdown 标题、编号、空行和无关标题进行规范化，随后按句号、分号等分隔符切分句子。对于句内复合操作，再按逗号、连接词和时间顺序词进一步拆分。

对于常见实验复合表达，方法采用显式展开规则。例如，“更换培养基”并不是单一自动化动作，而是可以分解为“吸去旧培养基”和“加入新鲜培养基”；“PBS 洗涤 3 次”可分解为“加入 PBS”“洗涤/混匀”“吸去 PBS”，并保留循环语义。这类规则的目的不是重写实验方案，而是将人工文本转化为装备控制和需求推导所需的原子操作单元。

每个操作随后通过动作词典映射到标准动作。词典记录动作类别、标准动作和别名，例如“加入、添加、滴加、补液”归一化为“加液”，“吸去、吸除、移除”归一化为“吸液”。别名按长度降序匹配，以降低短词过早命中的风险。

## 4. 字段抽取与专家规则推断

在每个原子操作内，AutoBiology 抽取或推断以下字段：动作、输入物、作用对象、容器、位置、工具、输出状态、参数、环境条件、人工判断、风险和处理方式。

参数抽取采用正则表达式和单位规范化。例如，温度识别 `4°C`，离心力识别 `5000g`，时间识别 `10 min` 或 `10 分钟`，体积识别 `5 mL`、`200 uL` 等。实体字段采用关键词和动作上下文推断，例如“细胞悬液”“细胞沉淀”“上清”“离心管”“培养瓶”等。

对于 SOP 未明确但工程建模必须记录的字段，系统采用专家规则标记来源。例如离心动作推断工具为冷冻离心机，转移、加液、吸液等液体处理动作推断存在污染风险；涉及“观察、确认、检查、是否、直至”等词汇时，标记为需要人工判断。此类补充被标注为 `expert` 或 `inferred`，并在超图中保留来源类型，避免与 SOP 明示证据混淆。

## 5. 知识超图建模

传统图边通常连接两个节点，而一个实验操作往往同时涉及动作、物料、目标、参数、环境、风险和处理方式。AutoBiology 因此采用操作超边建模：每个原子操作 `op_i` 生成一个超边 `e_i`，该超边连接与该操作有关的多类型节点。

节点类型包括 `Action`、`Precondition`、`Input`、`Target`、`Container`、`Location`、`Tool`、`OutputState`、`Parameter`、`Condition`、`HumanJudgment`、`Risk` 和 `Handling`。每个节点保留原始名称、规范名称、来源、来源操作集合和属性。节点复用通过键 `nodeType:normalizedName` 实现，使相同语义的节点在多个操作之间共享，从而减少重复并支持跨步骤追溯。

超边记录 `connectedNodes` 和按角色分组的 `nodeRoles`，同时记录 `missingInfo` 和 `expertSupplement`。其中 `missingInfo` 用于表示前置条件、输入物、作用对象或参数未充分说明等问题；`expertSupplement` 用于标记来源非 SOP 的补充节点。该设计使后续需求生成能够同时利用 SOP 证据和专家推断，但又不混淆两者的证据等级。

## 6. R1-R10 确定性需求生成

需求生成阶段以超图为输入，通过直接映射规则生成基础需求。R1-R10 分类对应不同工程问题：

- R1：操作功能需求，来自 `Action`、`Input`、`Target`、`OutputState`。
- R2：物料/容器兼容需求，来自 `Input`、`Target`、`Container`。
- R3：参数控制需求，来自 `Parameter`。
- R4：环境/约束需求，来自 `Condition` 和 `Risk`。
- R5：流程控制需求，来自 `ParentStep` 和 `Precondition`。
- R6：感知与判断需求，来自 `HumanJudgment` 和判断依据。
- R7：风险控制需求，来自 `Risk`。
- R8：异常处理需求，来自 `Risk` 和 `Handling`。
- R9：数据与追溯需求，来自操作、参数、风险和判断字段。
- R10：验证与验收需求，来自需求本身、参数、输出状态和风险。

这些规则的核心是证据字段到需求类型的确定性映射。每条生成的需求保留来源操作和来源超边，并记录 `inferenceRule`，如 `DM-R3` 表示由直接映射规则生成的 R3 参数控制需求。

除直接映射外，系统还实现领域模式检查。例如，离心操作通常需要温度、离心力和时间三个关键参数。若超边动作是“离心”但参数节点不包含上述字段，系统不直接生成正式参数需求，而是生成高优先级待澄清问题。这一设计符合“缺失信息不直接冻结为正式需求”的原则。

## 7. 需求去重与合并

需求去重采用指纹机制。每条需求的指纹由需求类型、规范化描述、排序后的来源字段和适用对象组成：

```text
fingerprint = SHA1(type | normalize(description) | sort(sourceFields) | normalize(applicableTo))
```

当新候选需求与已有需求指纹相同，系统合并其来源操作、来源超边、来源字段、关键指标、约束、相关风险，并保留较高置信度。该机制使多个 SOP 步骤产生同类需求时不会造成需求表膨胀，同时保留所有来源证据。

## 8. LLM 辅助候选推理

LLM 层是可选增强层，仅在 `apiKey`、`baseUrl` 和 `model` 均已配置时启用。其输入不是原始 SOP，而是确定性需求表。LLM 执行三个子任务：

1. 生成隐含候选需求。
2. 将候选需求改写为统一工程句式。
3. 判断候选需求与已有需求是否语义重复。

LLM 生成的候选必须包含 `source_hyperedge`。缺少来源超边的候选会被拒绝并记录待澄清项。通过该来源约束，LLM 不能产生无证据支撑的正式需求。所有 LLM 输出均标记为：

```json
{
  "sourceFields": ["LLM"],
  "responsibleModule": "llm-inference",
  "status": "candidate",
  "inferenceRule": "LLM-Candidate"
}
```

因此，LLM 输出需要经过人工或交互式审阅后才能转为确认状态。若 LLM 未配置或调用失败，系统保留确定性需求，并记录“LLM 辅助层未启用或已降级”的待澄清信息。

## 9. 覆盖率审阅与质量控制

审阅阶段构建超边-需求类型覆盖矩阵。矩阵行表示操作超边，列表示 R1-R10 需求类型，单元值为 `covered`、`missing` 或 `clarification`。覆盖率定义为：

```text
coverageRate = coveredCells / (numberOfHyperedges * numberOfRequirementTypes)
```

除了整体覆盖率，系统还执行角色覆盖检查。例如，存在 `Action` 节点但没有 R1 需求会生成警告；存在 `Parameter` 节点但没有 R3 需求会生成警告；存在 `Risk` 节点但没有 R7 需求会生成警告。该机制将需求完整性检查转化为可审计的矩阵计算。

审阅输出包括覆盖率 JSON、Markdown 报告和 Mermaid 图示，包括 SOP 流程图、超图关系图、需求追溯图、风险网络图和覆盖率图。交互式模式下，用户可对候选需求执行确认、拒绝、标记待澄清或跳过操作。

## 10. 可复现性与可解释性

AutoBiology 的中间产物均以 JSON 文件写出，包括 `01-ops.json`、`02-nodes.json`、`03-hyperedges.json`、`04-requirements.json`、`05-coverage.json`、`06-clarifications.json` 和 `run-meta.json`。这种设计使每个阶段的输入输出均可单独检查，也便于在论文复现实验中定位差异来源。

方法的可解释性来自三方面：

1. 结构化证据：需求保留 `sourceOps` 和 `sourceHyperedges`。
2. 推理规则标记：确定性需求记录 `DM-R*`，LLM 候选记录 `LLM-Candidate`。
3. 状态管理：候选、确认、待澄清、拒绝和冻结需求具有不同状态，避免将不确定推理误判为正式结论。

综上，AutoBiology 将 SOP 到装备设计需求的转换建模为一个可追溯的知识超图推理流程。其确定性算法层保证基础结果稳定可复现，LLM 辅助层提供隐含需求发现能力，而覆盖率与审阅层提供质量控制和专家闭环入口。
